# 生产环境数据库数据变更操作规范 (SOP)
## 1. 核心原则

● **先备份，后操作**：所有变更必须有回滚方案。
● **事务操作**：严禁在非事务模式下执行 UPDATE/DELETE。
● **双重校验**：预期影响行数与实际影响行数必须完全一致。
● **四眼原则**：脚本必须经过第二人（DBA或Team Leader）Review。

## 2. 操作流程详解

### 第一阶段：变更准备与评审 (Pre-Operation)
在执行任何变更前，必须编写标准的变更脚本（Change Script），包含以下三个部分：

**1. 确定预期影响范围（Scope Check）**
  a. 编写 SELECT COUNT(*) 语句，WHERE 条件必须与即将执行的 UPDATE/DELETE 完全一致。
  b. **目的**：获取一个具体的数字 （预期影响行数）。
  c. **检查点**：确认 WHERE 条件是否精准命中目标，防止全表更新。

**2. 数据备份/快照（Data Backup）**
  a. 编写 SELECT * 语句导出数据。
  b. **操作**：将受影响的原始数据导出为 Excel、CSV 或 SQL Insert 语句（作为回滚脚本）。
  c. **检查点**：如果业务逻辑改错了（虽然SQL语法是对的），可以通过此备份恢复原始值。

**3. 编写变更SQL**
  a. 确认字段名称（Field）无误。
  b. 确认更新的值（Value）符合业务逻辑。

### 第二阶段：执行过程 (Execution)

执行时必须严格按照以下事务三部曲进行操作。建议使用命令行或支持事务的客户端（如 DBeaver, Navicat, DataGrip）并将自动提交（Auto-Commit）关闭。

**步骤 1：开启事务**

```sql
START TRANSACTION; -- 或 BEGIN;
```

**步骤 2：执行变更**

```sql
-- 粘贴你的 UPDATE 或 DELETE 语句
UPDATE table_name SET status = 1 WHERE create_time < '2023-01-01';
```

**步骤 3：验证与提交/回滚 (关键步骤)**

● 查看执行结果：检查数据库返回的 `Rows matched` 或 `Affected rows`。
● 判定逻辑：
  
a. 情况 A： 实际影响行数 == 预期影响行数 N
    ⅰ. 👉 执行：COMMIT; (提交事务)
    ⅱ. ✅ 结果：变更生效。

b. 情况 B： 实际影响行数 != 预期影响行数 N (多了、少了、或者为0)
    ⅰ. 👉 执行：ROLLBACK; (立即回滚)
    ⅱ. ❌ 结果：操作取消，数据库无变化。需要重新排查 SQL 条件。

## 3. 标准操作脚本模板 (Template)

请要求开发人员按以下格式提交申请：
变更工单示例
目标：修正订单表 (t_order) 中 3 个异常订单的状态。预期影响行数：3 行

**1. 范围确认与备份 (执行前运行)**

```sql
-- 1.1 确认行数 (必须等于 3)
SELECT count(*) FROM t_order WHERE order_id IN (1001, 1002, 1003) AND status = 0;

-- 1.2 导出备份 (保存结果到本地文件)
SELECT * FROM t_order WHERE order_id IN (1001, 1002, 1003) AND status = 0;
```

**2. 变更执行 (复制到数据库终端执行)**

```sql
-- 2.1 开启事务
BEGIN;

-- 2.2 执行更新
UPDATE t_order
SET status = 1, update_time = NOW()
WHERE order_id IN (1001, 1002, 1003) AND status = 0;

-- 2.3 ！！！人工检查点！！！
-- 请观察控制台输出的 "Affected rows"。
-- 如果 Affected rows == 3，请执行 COMMIT;
-- 如果 Affected rows != 3，请执行 ROLLBACK;
```

**3. 验证语句 (提交后运行)**

```sql
SELECT order_id, status FROM t_order WHERE order_id IN (1001, 1002, 1003);
```

**4. 常见风险点 Checklist**

* WHERE 条件是否为空？ (严禁不带 WHERE 的更新)
* 是否包含 LIMIT？ (对于大批量数据修复，建议分批 LIMIT 1000 执行，防止锁表时间过长)
* ID是否唯一？ (如果通过非主键字段更新，可能会意外更新多条数据)
* 是否关闭了 Auto-Commit？ (防止敲完分号直接生效，无法回滚)
